# Web Mockups

Use this reference for web app screens, responsive web surfaces, and browser-framed product mockups.

## Artifact Types

- Web app workspace: show navigation, primary content, and secondary context only when useful.
- Dashboard: operational surface for monitoring, deciding, filtering, and acting.
- Detail or editor view: show the object, state, controls, and next action.
- Marketing page: only when the user asks for a site or landing page.
- Responsive comparison: show desktop and mobile side by side only when the task is about responsive behavior.

## Prompt Guidance

- State viewport target: desktop, tablet, mobile web, or responsive board.
- State whether browser chrome should appear.
- Name the product surface and user job.
- Include the key regions: navigation, workspace, inspector, filters, table/list, chart, preview, command bar, or form.
- Specify exact text for headings, buttons, and critical labels.
- Call out hierarchy: primary action, selected item, active state, empty state, or warning state.

## Web Defaults

- For product UI, default to dense but calm information design.
- Prefer plain layout, lists, tables, dividers, and grouped sections over decorative card grids.
- Avoid generic SaaS dashboards, oversized hero areas, badge spam, stock illustrations, and arbitrary gradients.
- Use cards only when the card is the interaction or object.
- Keep navigation predictable and scanning efficient.

## Comparison Boards

- Keep each option at the same viewport size.
- Label options outside the UI.
- If comparing one variable, hold shell, palette, type scale, and content constant.
- If exploring broadly, vary layout model, density, information grouping, and interaction emphasis.
