# Mobile Mockups

Use this reference for mobile app screens, mobile flows, and phone-framed concept boards.

## Artifact Types

- Single screen: one full phone screen, large enough to read.
- Flow board: 2-4 phone screens that show a coherent progression.
- Comparison board: A/B/C options with the same device, scale, and background.
- Native component context: a cropped screen section only when the component needs surrounding navigation or state.

## Prompt Guidance

- Name the platform: iOS, Android, or platform-neutral mobile.
- Specify the screen type: home, detail, editor, player, settings, search, modal sheet, empty state, error state, onboarding, or stateful component.
- Include required UI elements and exact text for labels, buttons, tabs, and headings.
- Describe safe areas, navigation treatment, bottom bars, sheets, keyboard state, and scroll behavior when relevant.
- Use a real device frame for full-screen mockups; keep the frame quiet and secondary.
- Keep the UI text large enough to inspect at the returned image size.

## Mobile Defaults

- Prefer native spacing, native controls, and platform-appropriate hierarchy.
- Use a restrained color system with one accent unless the repo design says otherwise.
- Avoid generic mobile app tropes: purple-blue gradients, floating glass cards, fake stats, meaningless icons, tiny labels, decorative blobs, and stock people.
- Do not make a landing page when the task is product UI.

## Device Framing

- Full iPhone screen: show the complete screen inside a device or clean mockup frame.
- Multiple iPhone screens: align screens evenly, keep labels outside the device, and use one consistent presentation style.
- Component-only task: avoid the full phone unless context is required.
- If the user asks to compare options, labels A/B/C should sit above or below the devices, not inside app navigation.
