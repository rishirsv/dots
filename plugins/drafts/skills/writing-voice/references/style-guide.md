# Style Guide Creation

A style guide captures the writing choices that should change future drafts. Do
not build a source audit. Read the samples like an editor.

## Start Light

If the user has no guide, ask for 2-3 things they wrote in this context, or do
a non-personalized pass now.

A few samples can steer the next draft. Keep the first result useful and
modest: a voice thesis, three to five observed choices, one or two examples,
and likely misses. Keep sample adequacy and caveats outside `style.md`.

Save a reusable guide only when the user asks to save, reuse, name, sync, or
maintain it.

## What To Notice

Look for repeated choices that change the writing:

- Sentence architecture: short punches, long explanatory lines, fragments,
  parallelism, subordination, sentence openers, and how clauses build.
- Paragraph rhythm: compact blocks, one-line turns, build-and-release,
  claim-example-implication, and list-heavy movement.
- Point of view: `I`, `we`, `you`, third person, direct address, distance from
  the reader, and when the writer appears or disappears.
- Argument movement: tension to thesis, story to lesson, evidence to meaning,
  objection to answer, diagnosis to fix, and insight to opportunity.
- Openings and endings: how the writing starts, lands, asks, closes, or hands
  off.
- Vocabulary: plain versus technical language, concrete versus abstract nouns,
  preferred verbs, recurring terms, taboo phrases, and level of polish.
- Punctuation and cadence: only when distinctive. Do not invent meanings for
  every mark.
- Signature moves: reusable habits that create the voice.
- Misses: what the style is likely to get wrong when imitated.

Prefer a few strong observations over a complete inventory. If a pattern would
not change the next draft, leave it out.

## What Is Not Voice

- Facts, sources, and topic vocabulary are content.
- Incoming messages, quoted text, signatures, boilerplate, and generated text
  are not the user's voice.
- House rules, brand mechanics, terminology, entity names, locale, and claims
  rules constrain the draft; they do not prove personal voice.
- Writing the user admires can set a target direction, but it is not evidence
  that the user already writes that way.
- Weak, duplicated, noisy, or mixed-author samples can support a session-only
  read. Do not turn them into a durable guide unless the user chooses that.

If context risks leaking into voice, say the boundary as a writing rule: "Use
current-task context as content, not voice."

## Write The Guide

Use `../assets/style.md`. Use `../examples/email-style.md` and
`../examples/brand-style.md` as examples.

Keep only what changes drafting:

- voice thesis
- moves that matter
- shape
- language
- examples
- avoid/fix
- optional modes
- revision checks

Do not include extraction notes, source inventories, ratings, sample counts,
storage paths, accepted/rejected reference lists, or instructions about how
strongly `compose` should rely on the guide.

## Corrections

Treat corrections as stronger than inference.

Promote a correction into `style.md` only when it is reusable. Keep one-off
preferences in the current draft or session. If the same miss recurs, add it to
the guide as an avoid/fix, revision check, or mode-specific rule.

If a correction conflicts with older observations, keep the guide modest until
more use resolves the drift.
