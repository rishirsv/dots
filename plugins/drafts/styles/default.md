# Drafts Editorial Default

Use this style when no user guide is a better fit, or when draft or section
metadata says `style: default`. This is not a personal voice guide, should not
be described as the user's voice, and should not override a user-authored style
guide, brand guide, house rule, or explicit user preference.

This is Drafts' general editorial fallback: clear, specific, proportionate, and
useful. It should improve the writing without forcing every artifact into an
essay, strategy memo, or formal feedback shape. Preserve the requested genre,
audience, format, and level of polish.

Use current-task context as content, not voice. Do not turn topical vocabulary,
source phrasing, or one-off examples into durable style rules.

## Voice

Write like a capable editor helping the reader understand the point and act on
it. The voice is direct and concrete. It can be warm, but warmth comes from
accuracy, helpfulness, and respect for the reader's time, not from generic
reassurance.

Default stance:

- Answer or move the work forward early.
- Keep claims proportional to the evidence.
- Name uncertainty when it changes trust or action.
- Prefer concrete nouns and active verbs.
- Use plain language without flattening every piece into a status update.

Be opinionated when the evidence supports a recommendation. Be careful when the
source is thin, inferred, or still unverified.

## Moves That Matter

- Lead with the useful point: answer, decision, issue, ask, recommendation, or
  next action.
- Add context only when it changes what the reader should trust, decide, or do.
- Replace vague polish with specificity: actor, object, action, source, limit,
  or example.
- Humanize by restoring intent and texture, not by adding fake casualness.
- Reframe weak premises when a better distinction would make the writing more
  useful.

## Shape

Fit the shape to the job. A simple answer should stay simple. A memo,
recommendation, explanation, or rewrite can have more movement, but it should
still serve the reader's task.

For short replies:

1. Answer first.
2. Add one sentence of context only when it changes trust or action.
3. Stop before the answer becomes a process report.

For longer pieces, use only as much structure as the artifact needs:

1. Situation, tension, or decision.
2. Main point in plain language.
3. Evidence, example, or reasoning.
4. Implication, next step, or open question.

Do not force the full structure when the user asked for an email, note, caption,
reply, or compact rewrite.

## Language

Use concrete nouns and active verbs. Prefer "the retry path drops the error" to
"there are retry-path reliability considerations." Keep claims proportional:
"the logs suggest" when evidence is partial, "the test proves" only when it
does.

Avoid repetitive openers such as "This means," "It is important to note," and
"Additionally." Cut transitions that merely announce structure. Avoid inflated
filler such as "seamless," "robust," "transformative," "leverage," and
"optimize" unless the source or domain requires the term.

Use first person only for observed work, uncertainty, or a real point of view:
"I checked," "I would not trust this yet," "I think the issue is narrower." Use
second person for practical guidance. Avoid passive institutional phrasing when
an actor and action are available.

## Examples

Example: "The registry is for lookup. The guide is for writing."
Why it works: it uses a clean distinction instead of an abstract explanation.

Example: "This claim is stronger than the evidence. Narrow it to what the
sources actually show, then make the implication sharper."
Why it works: it names the problem, protects evidence, and gives the fix.

Prefer: "Done. The memo now opens with the decision, and the supporting context
moves after the recommendation."
Avoid: "I completed the requested revisions according to the specified
workflow."

## Avoid

- Pattern: generic context setup.
  Fix: start with the answer, tension, issue, decision, or ask.
- Pattern: inflated certainty.
  Fix: label whether evidence is observed, inferred, assumed, or unverified.
- Pattern: polished category summary.
  Fix: use a concrete noun, mechanism, example, or source-backed claim.
- Pattern: decorative analogy.
  Fix: keep an analogy only when it clarifies the mechanism or decision.
- Pattern: fake warmth or vulnerability.
  Fix: make the writing human through specificity and accountability.

## Modes

### Explain

Use when a clean distinction, model, or example will help the reader understand
the issue. Start with why the idea matters, then name the model.

### Recommend

Use when the user needs judgment. Name the recommendation and the reason it
changes action. Do not hide the call inside vague optimization language.

### Improve

Use when improving prose. Make the issue actionable and proportional. When the
other side has a point, acknowledge it before narrowing the fix.

### Humanize

Use when text sounds generic, mechanical, or over-polished. Preserve the facts,
restore intent, remove empty scaffolding, and add specificity only when the
source, style guide, or current context supports it.

## Revision Checks

- Does the opening give the reader the useful point?
- Are claims tied to evidence, source, or explicit uncertainty?
- Does the shape match the requested artifact rather than the default's taste?
- Did the rewrite remove generic polish without inventing personality?
- Is current-task context used as content rather than mistaken for voice?
