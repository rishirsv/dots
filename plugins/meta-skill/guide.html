The mental model

Two folders, one rule.

A skill is a portable payload you can hand to any agent. Everything that helps you build and test it lives beside it in a .meta-skill/ workbench that never ships. The rule: improve only from evidence.

my-skill/
├─ SKILL.md          ← the skill (always ships)
├─ references/        ← deep detail, loaded on demand
├─ scripts/  agents/  assets/
└─ .meta-skill/       ← workbench (never packages)
   ├─ spec.md          decisions + residual risk
   ├─ evals/           scenarios/ judges/ runs/
   ├─ tests/           manifest.json
   ├─ reviews/ plans/ sessions/
   └─ versions/release/ ← sealed snapshot

Portable payload

The root. What an agent runs, and the only thing package exports.

.meta-skill/ workbench

Added on demand with project init. Holds every spec, eval run, and decision as inspectable evidence — and stays out of the shipped skill.

Portable by default

Start with just SKILL.md. Add the workbench only when you need evals, releases, or team reuse.

The three lanes

Each lane is a skill and a set of commands.

You stay in plain language — the plugin routes your intent to the right lane and runs the CLI for stable file actions. Steps marked with a lock need your approval first.

Create

 skill-create

Shape a workflow into a portable skill: a sharp trigger, clear boundaries, and the smallest useful runtime. Lint validates the frontmatter and payload contract.

  1. create
  2. project init optional
  3. lint

Produces · a valid SKILL.md + payload, structure and frontmatter checked.

Evaluate

 skill-eval

Run the skill against real scenarios through App Server in a read-only sandbox, and collect evidence from final answers, traces, deterministic tests, and optional judges.

  1. eval init
  2. eval run
  3. eval judge optional
  4. eval open

Produces · a run under evals/runs/ with final answers, traces, usage, and a report.

Improve

 skill-improve

Read the evidence, plan one bounded edit, apply it to the payload, and record the decision. Review scores eight best-practice dimensions.

  1. review
  2. plan
  3. promote
  4. decide

Produces · a plan tied to a run or review, a promoted edit, and a recorded decision in sessions/.

Ship

 release · package

Seal a validated snapshot, then export the portable payload as a zip. Both wait for an explicit go and require a clean lint.

  1. release
  2. package

Produces · a sealed versions/release/ and a shippable zip of the payload only.

Inside an eval

Evidence you can read, not a score you trust blindly.

Scenarios are tagged by what they stress. Each run executes through App Server and writes final-answer and trace evidence to disk.

Scenario families

R
Regression

A normal task that should keep working.

F
Failure mode

A hard, ambiguous, or multi-turn case.

T
Trigger

Activates — or correctly declines — on the boundary.

G
Gate

Stops for approval or a safe default.

read-only no network no auto-approval

What a run records

.meta-skill/evals/runs/003-after-edit/
run.json orchestration + sandbox config events.jsonl run lifecycle results.jsonl per-scenario outcome + tokens tests.jsonl deterministic test results grades.jsonl judge scorecards (optional) report.html the readable summary scenarios/F1/ final.md the agent's answer rpc.jsonl full turn-by-turn trace

Token usage is recorded per scenario — or marked unavailable with a reason when App Server returns none.

Human gates

Nothing leaves without a yes.

The plugin moves fast inside the workbench, but stops at every edge where work becomes real. You confirm before anything is written into source or sent outward.

Into source

  • promote writes a planned edit into the live payload
  • only applies edits from a plan you approved

Sealing a version

  • release snapshots a validated build
  • blocked unless lint passes; confirms before replacing

Going outward

  • package, install, publish, or marketplace sync
  • judges cost tokens — opt in with --with-judges

Command reference

One CLI, grouped by lane.

Every command takes the project path. The agent narrates intent; these are the stable actions it runs.

Evaluate
eval initscaffold the evals workbench and scan for scenarios

eval judgescore saved evidence with LLM judges (optional, costs tokens)
eval openopen a run's report.html, or list runs
Improve
reviewscore eight best-practice dimensions into a review

deciderecord an accept or reject decision for the session
Ship
release seal a validated snapshot under versions/release/
package export the portable payload as a zip (payload only)

Quickstart

Your first run, top to bottom.

1

Create a portable skill

One folder, one SKILL.md. Front-load the trigger.

2

Add a workbench

Only when you want evals, releases, or reuse.

3

Lint & review

Catch structure problems and best-practice gaps.

4

Evaluate

Add starter scenarios, run them, open the report.

5

Improve from a run

Plan one edit tied to the run, promote it, decide.

first-run.sh
# 1 · create a portable skill $ meta-skill create my-skill --slug my-skill --description "Use when …; not for …" $ cd my-skill # 2 · add a workbench (optional) $ meta-skill project init . # 3 · check & review $ meta-skill lint . $ meta-skill review . # 4 · scenario evals $ meta-skill eval init . $ meta-skill eval run . $ meta-skill eval open . # 5 · improve from the run $ meta-skill plan . --from-run <run-id> $ meta-skill promote . --plan <plan-id> $ meta-skill decide . --session <session-id> --accept

The eval lane runs through Codex App Server, so it needs the codex CLI installed and authenticated. Everything else runs locally with no dependencies.
