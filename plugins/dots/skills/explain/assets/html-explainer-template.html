Idempotency keys stop a double charge on retry.

When a checkout request times out and the client retries, the same key short-circuits to the first response instead of running the charge twice. Here is the path, where the key is stored, and the one race that still bites.

The request path

flow

Solid = the request path. Dashed green = the retry that reuses the stored result.

  1. ClientCheckout retries

    The same idempotency key returns with the second request.

  2. HandlerLookup first

    The handler checks for a saved response before calling the provider.

  3. GapConcurrent first call

    Two initial requests can both miss before either commit exists.

  4. StoreKey row

    The response is stored for the retry window.

On a retry the handler finds the key row and returns the saved response. The PSP is never called twice.
