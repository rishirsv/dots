# Meta Skill Report HTML Shell V1 Spec

Meta Skill needs a small local web app that opens as a saved `report.html` in Codex Browser. The first version should make a completed eval run visually inspectable without adding a server, framework, live mode, artifact viewer, or new runner behavior.

## Product Shape

V1 is a per-run inspector.

The user opens one run and answers:

- Which scenarios need attention?
- What did candidate and release produce?
- Did the output pass tests, judges, or human feedback?
- What token usage was recorded?
- Where is the raw evidence if I need to audit it?

The future product can add a skill dashboard across all runs and variants. V1 should still use stable concepts that support that future:

- Skill: the reusable skill project being evaluated.
- Variant: the evaluated skill payload, initially `candidate` and `release`.
- Scenario: one eval prompt/task.
- Attempt: one variant's execution for one scenario in one run.
- Run: one batch execution across scenarios and variants.

## Non-Goals

- No artifact viewer in V1.
- No live watch mode.
- No local web server.
- No React, Vite, router, or build pipeline.
- No pricing model.
- No trigger-routing or no-skill uplift proof.
- No arbitrary variant execution beyond the current candidate/release runner.
- No editable skill tweaks inside the report.

If a scenario is artifact-oriented, V1 still shows final-answer evidence, tests, judges, feedback, and raw links. It does not render generated artifacts.

## Tech Stack

Use a static-first local app generated by the Meta Skill CLI:

- Renderer: TypeScript functions in `plugins/meta-skill/src/report.ts`.
- Output: one self-contained `report.html`.
- Data: normalized report payload embedded as JSON in `<script type="application/json">`.
- Interaction: small inline JavaScript for selecting scenarios, filtering, and toggling detail panels.
- Styling: inline CSS with system fonts, stable dimensions, restrained color, semantic status accents, and no external assets.
- Raw evidence: normal relative links to `report.json`, `turns.jsonl`, `usage.json`, `rpc.jsonl`, tests, judges, feedback, and final outputs.

The Codex App Server remains the evidence producer. Meta Skill remains the normalizer. The browser app reads saved evidence; it does not talk to the App Server.

## V1 Data Model

The app can be generated from existing `RunReport`, but the rendered app payload should be shaped for UI use:

```ts
interface ReportAppModelV1 {
  skill: {
    name: string;
    project_path?: string;
  };
  run: {
    id: string;
    label?: string | null;
    status: string;
    assessment_status: "passed" | "needs_review" | "failed" | "unknown";
    readiness_status: "ready" | "needs_review" | "blocked";
    readiness_summary: string;
    comparison_mode: "none" | "release";
    manual_review_required: boolean;
    created_at?: string;
    completed_at?: string;
    runner_backend?: string;
    runner_mode?: string;
  };
  variants: Array<{
    id: "candidate" | "release";
    label: string;
    kind: "working_tree" | "release";
  }>;
  summary: {
    scenario_count: number;
    unresolved_count: number;
    failure_classifications: string[];
    token_usage: RunTokenUsageSummary;
  };
  scenarios: Array<{
    id: string;
    folder: string;
    title: string;
    family?: string;
    type?: string;
    topics: string[];
    capability?: string | null;
    status: string;
    unresolved: boolean;
    evidence_basis: string;
    criteria?: {
      what_it_tests?: string;
      expected_behavior?: string;
      assertions: string[];
      tests: string[];
      judges: string[];
    };
    attempts: Array<{
      variant_id: "candidate" | "release";
      status: string;
      final_preview?: string;
      final_path?: string;
      evidence_path: string;
      token_usage?: TokenUsageSummary;
      failure_classification?: string | null;
      error?: string;
    }>;
    comparison?: {
      kind: "release" | "none";
      classification: string;
    };
    evidence: {
      final_links: Array<{ label: string; href: string }>;
      raw_links: Array<{ label: string; href: string }>;
      tests: EventEnvelope[];
      judges: EventEnvelope[];
      feedback: EventEnvelope[];
    };
  }>;
}
```

V1 should not change the canonical evidence files. It may introduce a UI-specific adapter inside the renderer to avoid making the HTML template understand every legacy shape.

## UX Layout

Use a dense product UI, not a marketing page.

Primary layout:

- Sticky top bar: skill name, run id, run status, readiness, open raw report link.
- Left scenario rail: scenario search/filter, status chips, scenario list with compact badges.
- Main workspace: selected scenario, candidate/release output comparison, evaluation evidence.
- Right inspector: run context, token usage, criteria, evidence links.

Default selected scenario:

1. First failed scenario.
2. Else first `needs_review` scenario.
3. Else first scenario.

Filters:

- All
- Needs review
- Failed
- Passed
- Changed
- Token unavailable

The filter bar should reduce the scenario rail only. It should not hide run-level summary evidence.

## Minimum Reporting Sections

V1 should include:

- Run status and readiness.
- Manual review requirement.
- Forced-skill evidence label.
- Scenario status counts.
- Candidate and release final-answer previews.
- Candidate/release comparison classification.
- Token usage by variant and selected scenario.
- Tests summary and rows.
- Judge summary and rows.
- Feedback summary and rows.
- Scenario criteria and expected behavior.
- Raw evidence links.

V1 should exclude:

- Artifact previews.
- Full transcript rendering.
- Rich diff algorithm.
- Cross-run trend charts.
- Editable feedback forms.
- Skill editing or tweak application.

## Comparison Behavior

If both candidate and release exist:

- Show two output panes side by side.
- Keep status and token usage attached to each side.
- Show comparison classification as evidence, not as a verdict by itself.
- Never pool candidate and release tokens.

If only candidate exists:

- Show candidate output full width.
- Replace release pane with a quiet empty state: `No release attempt in this run.`

If final output is missing:

- Show status, error, and raw evidence links.
- Do not invent a preview.

## Visual Direction

Register: product, local evaluation workbench.

Theme scene: a skill author is reviewing eval evidence on a laptop inside Codex after a run finishes, trying to decide which scenario deserves the next edit.

Color strategy: restrained neutral UI with semantic accents.

Visual system:

- System sans font stack.
- Hairline borders and tonal layers, not heavy shadows.
- Radius family: 6px for panels and controls.
- Stable tabular numbers for counts and tokens.
- Accent only for selected scenario, active filter, and status.
- Compact rows and readable line lengths.

## Acceptance Signals

The first implementation is successful when:

- `report.html` opens directly in Codex Browser from the filesystem.
- A user can identify the next scenario to inspect within 10 seconds.
- A user can compare candidate and release final outputs without opening raw files.
- Token usage is visible but never framed as a quality score.
- `needs_review` is visibly unresolved, not pass proof.
- Forced-skill evidence is labeled.
- Raw evidence links remain available.
- Legacy/minimal reports render without throwing.
- The app remains self-contained and small enough to be generated on every run.

## Implementation Notes

Build this as a renderer refactor, not a reporting-system rewrite:

1. Add a UI adapter that converts `RunReport` to `ReportAppModelV1`.
2. Render static shell markup with landmarks and empty states.
3. Embed app data as escaped JSON.
4. Add small inline JS for scenario selection and filters.
5. Keep existing report summary table behavior covered by tests until the new shell has equivalent assertions.
6. Add tests for shell landmarks, legacy token shape, candidate-only, release comparison, token unavailable, and no scenario rows.
7. Inspect generated HTML in Codex Browser before treating the slice as done.
